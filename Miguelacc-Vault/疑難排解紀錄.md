## node_modules 安裝不完整

> [!bug] 症狀
> - `node_modules/.bin` 為空，`prisma` / `next` 指令無法執行
> - 首頁 500：`Cannot find module '.../magic-string/dist/magic-string.cjs.js'`（套件 dist 目錄是空的）

**原因**：node_modules 被部分複製 / 安裝不完整，bin 捷徑與部分套件內容缺失。

**解法**：
```bash
npm ci          # 從 lockfile 完整重裝（移除並重建 node_modules）
npx prisma generate
```

## 跨來源 cross-origin 導致按鈕沒反應

> [!bug] 症狀
> 用 `http://127.0.0.1:3000` 開網頁時「加入購物車」按鈕點了沒反應。
> 伺服器 log：`⚠ Blocked cross-origin request to Next.js dev resource /_next/webpack-hmr from "127.0.0.1"`

**原因**：Next.js 16 開發模式預設只信任 `localhost`，把 `127.0.0.1` 的 client 端 dev 資源擋掉，導致 client component 沒能 hydration，事件處理器沒掛上。

**解法**：在 `next.config.ts` 加入白名單：
```ts
const nextConfig: NextConfig = {
  allowedDevOrigins: ["127.0.0.1", "10.46.192.96"],
  // ...
};
```
> [!tip] 更省事的做法：直接用 `http://localhost:3000` 開網頁即可避免。

## 結帳頁無限迴圈 Maximum update depth exceeded

> [!bug] 症狀
> 進到 `/checkout` 後跳錯：`Maximum update depth exceeded`，指向 `cart-provider.tsx` 的 `clear`。

**原因**：`cart-provider.tsx` 的 `clear`（及其他函式）每次 render 都建立新參照。結帳頁 `useEffect` 依賴 `[state, clear]`：`clear()` → `setItems([])` → 重新 render → `clear` 變新參照 → effect 又執行 → 無限迴圈。

**解法**：在 `src/components/cart-provider.tsx` 用 `useCallback` 穩定所有函式，並把 `value` 用 `useMemo` 包起來：
```ts
const clear = useCallback(
  () => setItems((prev) => (prev.length === 0 ? prev : [])),
  []
);
```

## 綠界「頁面暫時無法顯示」— 付款端點 URL 用錯（真正主因）

> [!bug] 症狀
> 結帳跳轉到綠界後出現：「很抱歉，頁面暫時無法顯示，請重新整理或稍後再試!」

**排查過程**（用瀏覽器自動化逐項排除）：
1. ✅ CheckMacValue 演算法 → 用綠界官方測試向量驗證，hash 完全相符（**正確**）
2. ✅ 換官方測試憑證 `3002607` → 仍失敗（排除憑證）
3. ✅ 換公開網址（非 localhost）→ 仍失敗（排除 localhost）
4. 🎯 最後發現：**付款端點 URL 是舊的**

**真正原因**：`src/lib/ecpay.ts` 的付款網址用了舊端點 `/Checkout/AioCheckout`，綠界現行端點是 `/Cashier/AioCheckOut/V5`。

**解法**：
```ts
export const ECPAY_PAYMENT_URL = {
  prod: "https://payment.ecpay.com.tw/Cashier/AioCheckOut/V5",
  stage: "https://payment-stage.ecpay.com.tw/Cashier/AioCheckOut/V5",
} as const;
```
修正後付款頁正常顯示（信用卡、Apple Pay、ATM、超商…）。

## 綠界錯誤碼 10200074「找不到加密金鑰」

> [!bug] 症狀
> 端點修正後，用自己的商店憑證 `3385459` 出現「交易失敗，訊息代碼 10200074，找不到加密金鑰」。

**原因**：`3385459` 是**正式環境**的商店憑證，但程式設定為測試環境（`ECPAY_IS_STAGE="true"`、打 `payment-stage`）。測試與正式是分開的兩套系統，正式憑證在測試環境不存在。

**解法**：本機測試改用綠界官方公開測試憑證 `3002607`（見 [[綠界測試金流]]）。正式憑證 `3385459` 只在上線時搭配 `ECPAY_IS_STAGE="false"` 使用。

> [!note] CheckMacValue 修正仍保留
> 排查過程中也修掉一個 `computeCheckMacValue` 的潛在 bug（誤編碼 `! ( ) *`，.NET 視為安全字元不編碼）。雖非本次主因，但商品名稱含半形括號時會出錯，已一併修正為符合 .NET 行為的版本。
